## 1. System at a Glance

Sift is an AI-powered web research tool designed for personal knowledge workflows. It searches the web via DuckDuckGo, extracts content from pages using trafilatura, indexes everything in a local SQLite database with FTS5 full-text search, and synthesizes answers using LLM APIs with inline citations. The system follows a local-first philosophy: all data lives on the user's machine in `~/.sift/`, with optional export to an LLM wiki at `~/llm-wiki/`. The architecture is a modular Python CLI application with clear separation between data ingestion (feeds/pulse), storage (SQLite/FTS5), and synthesis (LLM APIs).

## 3. Component Map

### CLI Entry Point
- **Type:** CLI application entry point
- **Location:** `sift/cli.py`
- **Responsibility:** Parses commands, initializes DB context, routes to subcommands
- **Entry point:** `sift/cli.py:main()` (registered as `sift` console script)
- **Key dependencies:** Click, `sift.db.DB`

### Database Layer
- **Type:** Data persistence library
- **Location:** `sift/db.py`
- **Responsibility:** SQLite schema management, FTS5 search, CRUD operations
- **Entry point:** `sift/db.py:DB` class
- **Key dependencies:** sqlite3 (stdlib), WAL mode enabled for concurrency

### Feed Ingestion Engine
- **Type:** RSS/Atom feed fetcher
- **Location:** `sift/feeds.py`
- **Responsibility:** Fetches feeds, parses XML, extracts page content, stores to DB
- **Entry point:** `sift/feeds.py:FeedFetcher`
- **Key dependencies:** requests, trafilatura, xml.etree.ElementTree

### Pulse Research Engine
- **Type:** Recursive web research crawler
- **Location:** `sift/pulse.py`
- **Responsibility:** DDG search, query variation generation, depth-limited link crawling
- **Entry point:** `sift/pulse.py:PulseEngine`
- **Key dependencies:** ddgs (DuckDuckGo Search), requests, trafilatura, html.parser

### Domain Crawler
- **Type:** Sitemap-aware domain spider
- **Location:** `sift/crawler.py`
- **Responsibility:** Discovers sitemaps, parses XML sitemaps, BFS fallback crawling
- **Entry point:** `sift/crawler.py:DomainCrawler`
- **Key dependencies:** requests, trafilatura, xml.etree.ElementTree

### LLM Synthesis
- **Type:** Answer generation with citations
- **Location:** `sift/synthesize.py`
- **Responsibility:** Builds context from search results, streams LLM responses
- **Entry point:** `sift/synthesize.py:synthesize_stream()`
- **Key dependencies:** httpx (streaming), python-dotenv (API key resolution)

### Wiki Export
- **Type:** Research output formatter
- **Location:** `sift/wiki.py`
- **Responsibility:** Formats synthesis for wiki, splits reasoning from answer
- **Entry point:** `sift/wiki.py:write_raw_source()`
- **Key dependencies:** pathlib (stdlib), datetime (stdlib)

### Test Suite
- **Type:** Unit tests
- **Location:** `tests/`
- **Responsibility:** Validates crawler, DB, feeds, pulse, and synthesis modules

## 4. Data Flow

### Primary Research Flow (ask command)

```
User Query
    ↓
[sift ask] → Search local FTS5 index (sift.db)
    ↓ (if no results)
[PulseEngine] → DDG search with query variations → Rank by frequency
    ↓
Fetch top 15 pages → trafilatura extraction → Store in pages table
    ↓
[depth=2] Extract links from top 5 pages → Fetch depth-1 links
    ↓
Build context from results (title + URL + excerpt)
    ↓
[synthesize_stream] → POST to OpenAI-compatible API (streaming)
    ↓
Display answer with inline citations [1], [2], etc.
    ↓
[--wiki flag] → Format with YAML frontmatter → Write to ~/llm-wiki/raw/queries/
```

### Feed Ingestion Flow

```
[sift ingest] → Load all sources from DB
    ↓
For each feed: fetch_feed() → Parse RSS/Atom XML
    ↓
For each entry: Check if URL exists in pages table
    ↓ (if new)
fetch_page() → trafilatura extraction → add_page()
    ↓
Sleep 1s between fetches (rate limiting)
```

### Domain Crawl Flow

```
[sift crawl <url>] → _get_root() extracts scheme://hostname
    ↓
_discover_sitemaps() → Try /robots.txt → Parse Sitemap: directives
    ↓ (if no sitemaps found)
_crawl_from_root() → BFS from root following internal links
    ↓
_parse_sitemap() or _crawl_from_root() → List of URLs
    ↓
For each URL: fetch → trafilatura → add_page()
    ↓
Sleep 0.5s between fetches
```

## 5. Service Boundaries

Sift is a single-process CLI application with no network services. All boundaries are internal module boundaries:

| Component | Communication Method | Shared State |
|-----------|---------------------|--------------|
| CLI ↔ All modules | Python function calls | `ctx.obj["db"]` context object |
| Database ↔ All modules | SQLite connection | `~/.sift/sift.db` file |
| FeedFetcher ↔ PulseEngine | No direct communication | Both write to `pages` table |
| Synthesis ↔ Wiki | Function calls | In-memory strings |
| Tests ↔ Production code | Import and call | Temporary databases via `tmp_path` |

**Shared Infrastructure:**
- **SQLite Database:** `~/.sift/sift.db` — shared by all commands
- **FTS5 Virtual Table:** `pages_fts` — updated via triggers on pages table
- **HTTP Session:** `requests.Session` with shared User-Agent and headers
- **API Keys:** Environment variables or `.env` file (resolved at import time)

## 6. Key Design Decisions

- **Local-first data ownership** — All content stored in `~/.sift/sift.db` on user's machine; no cloud dependency for core functionality. Rationale: Personal research tool philosophy.

- **FTS5 with Porter stemming** — Virtual table `pages_fts` uses `tokenize='porter unicode61'` for English text search with stemming. Evidence: `sift/db.py` line 47.

- **WAL mode for SQLite** — `PRAGMA journal_mode=WAL` enables concurrent reads during writes. Evidence: `sift/db.py` line 17.

- **Query variation generation** — Pulse generates 8 query variations (exact, quoted, "what is X", "X research 2026", etc.) to maximize result diversity. Evidence: `sift/pulse.py` lines 80-92.

- **Frequency-based URL ranking** — URLs appearing in multiple variation results are ranked higher. Evidence: `sift/pulse.py` lines 257-276.

- **Depth-limited crawling** — Pulse fetches 15 pages at depth 0, then follows links from top 5 to depth 1. Evidence: `sift/pulse.py` lines 278-309.

- **Streaming LLM responses** — `synthesize_stream()` yields tokens as they arrive via SSE, providing UX responsiveness. Evidence: `sift/synthesize.py` lines 158-244.

- **Rate limiting via sleep** — 0.5s between page fetches in pulse/crawler, 1s in feeds. Evidence: `sift/pulse.py` line 289, `sift/feeds.py` line 245.

- **Social media domain filtering** — Facebook, Twitter/X, LinkedIn, Instagram URLs are excluded from link extraction. Evidence: `sift/pulse.py` lines 50-61.

- **Content quality threshold** — Pages with <50 characters of extracted text are discarded. Evidence: `sift/pulse.py` line 179, `sift/crawler.py` line 302.

- **Reasoning/answer split** — Wiki output separates model reasoning from final answer using regex heuristics. Evidence: `sift/wiki.py` lines 18-75.

## 7. What's Not Covered Here

The following areas are underdocumented or could not be fully resolved from code inspection:

- **API key rotation strategy** — No mechanism for handling expired keys or fallback providers beyond environment variable precedence.

- **Error recovery from partial pulses** — If a pulse fails mid-run, there's no resume mechanism; partial results remain in DB but pulse record shows incomplete state.

- **FTS5 query syntax edge cases** — The query sanitization in `db.py` (line 108) quotes special characters, but complex boolean queries are not documented.

- **Wiki curation workflow** — Raw sources are written to `~/llm-wiki/raw/queries/`, but the path from raw → curated concepts is not automated.

- **Feed deduplication strategy** — Feeds use URL as unique key, but no content-hash deduplication for pages that change URLs.

- **Database migration strategy** — Schema is created via `executescript()` in `_init_schema()`; no versioned migrations for schema changes.
